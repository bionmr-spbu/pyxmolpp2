Frame/Chain/Residue/Atom hierarchy¶

The pyxmolpp2 library implements Frame/Chain/Residue/Atom hierarchy to represent a molecular system.

Every Atom exists as a part of some Residue. Residue is always a part of Chain. Chain is always a part of Frame.

For instance this implies that you can not create an Atom without pre-existed Residue. Why not allow Atom.residue to be None if Atom exists by it’s own? The choice was between flexibility of construction code versus complexity of it’s further usage. The rationale behind that design decision is fact that expression atom.residue.chain.frame.index is always correct, and no not-None checks are required. Such checks across the library and user code would increases it’s complexity and make it more prone to errors.

Danger

Atom/Residue/Chain is alive until it’s Frame exists. Frame exists until python holds a reference to it. No frame - no game.
